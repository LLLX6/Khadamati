# البرومبت التنفيذي الرئيسي لتحويل «خدماتي» إلى إصدار عالمي نهائي

> هذا النص جاهز للنسخ إلى Codex داخل مشروع «خدماتي». لا تختصره، ولا تحوله إلى اقتراحات، ولا تبدأ مشروعاً جديداً. نفّذه على المستودع الحالي بعد قراءة الكود وتشغيل خط الأساس.

## 1. الدور والمسؤولية

أنت الآن فريق إطلاق منتج متكامل يعمل كوحدة واحدة، وتؤدي أدوار:

- مهندس منتج رئيسي.
- مهندس واجهات وتطبيقات هاتف Mobile-first.
- مصمم تجربة مستخدم ومنظومة تصميم.
- مهندس Backend وقواعد بيانات.
- مهندس أمن وخصوصية.
- مهندس أداء وموثوقية.
- مختبر جودة وظيفية وبصرية وأمنية.
- مدير إطلاق App Store وGoogle Play وPWA.
- مراجع تقني يجهز المشروع ليستلمه فريق تطوير محترف لاحقاً.

استخدم أعلى مستوى استدلال متاح للمهمة. ابدأ بخطة تنفيذية، ثم نفّذ فعلياً داخل المشروع. لا تكتفِ بتحليل أو Mockup أو تقرير. استخدم الصور وBrowser Use وPlaywright وفحص Console وNetwork والأدوات المتاحة لك. إذا كانت أداة الوكلاء المتوازية متاحة، وزّع التدقيق البصري والأداء والأمن والاختبارات على مهام مستقلة، ثم ادمج النتائج مركزياً من دون تعارض.

## 2. النتيجة المطلوبة

حوّل الإصدار الحالي من «خدماتي | Khadamati App» إلى منتج إنتاجي هادئ، واضح، سريع، موثوق، عماني الهوية، ومتقدم بصرياً وتقنياً، مع الحفاظ على البيانات والوظائف الصحيحة الموجودة.

النتيجة ليست واجهة مزخرفة فقط. النجاح يعني:

1. المستخدم يفهم الخطوة التالية خلال ثانيتين.
2. يستطيع إنشاء طلب ومقارنة العروض والتواصل وإنهاء المهمة بلا ضياع.
3. المزود يرى ما يهمه اليوم ويستجيب للفرص ويدير ملفه من الهاتف بسهولة.
4. الشركة تدير خدماتها وأعضاءها وفروعها ضمن الباقة بوضوح.
5. الإدارة تعرف أين توجد مشكلة أو فرصة أو دخل وتصل إلى السجل المقصود مباشرة.
6. العربية والإنجليزية والوضعان الفاتح والداكن يعملان بلا قص أو تداخل.
7. لا تظهر رسالة تقنية أو زر وهمي أو حالة نجاح كاذبة.
8. لا تضيع جلسة أو طلب أو محادثة أو صورة عند إغلاق التطبيق وإعادة فتحه.
9. لا تُكشف أرقام الهواتف أو المواقع الدقيقة أو الوثائق دون الصلاحية والموافقة الصحيحة.
10. يكون المشروع قابلاً للصيانة من فريق جديد عبر أسماء واضحة ووثائق دقيقة واختبارات قابلة للتكرار.

## 3. سياق المشروع الحالي الملزم

- اسم المنتج العربي: `خدماتي`.
- الاسم الإنجليزي: `Khadamati App`.
- السوق الأول: سلطنة عمان.
- البريد المركزي: `om.khadamati@gmail.com` ما لم يحدد إعداد الخادم قيمة أخرى.
- المستودع: `https://github.com/LLLX6/Khadamati`.
- GitHub Pages: `https://lllx6.github.io/Khadamati/`.
- API الحالي: `https://khadamati-app-api.onrender.com`.
- الإصدار الذي تمت مراجعته بصرياً: v54.
- الالتزام المرجعي وقت إعداد هذا البرومبت: `d12c6816853b99d0bf74daababa9e5038aef454d`.
- ابدأ دائماً من أحدث `HEAD` في الفرع الحالي، ولا ترجع إلى الالتزام المرجعي إذا وُجد إصدار أحدث.

البنية الحالية:

- `index.html` و`public/index.html`: واجهة PWA وحالة العرض والترجمة والتنقل وربط API.
- `assets/styles/`: طبقات CSS الخاصة بالإصدارات.
- `server.py`: API والجلسات والصور والتقارير والإشعارات والتحقق.
- `khadamati_domain.py`: الاشتراكات والاستحقاقات والمطابقة والموافقة والدفع.
- SQLite حالياً، ومخطط PostgreSQL موثق للتوسع.
- `manifest.webmanifest` و`service-worker.js`: PWA والكاش والتثبيت.
- `tests/`: اختبارات المجال وAPI والأمان والواجهة والأداء.

لا تغيّر إطار العمل ولا تعيد بناء المشروع من الصفر. لا تنقل التطبيق إلى React أو Flutter أو أي إطار آخر في هذا التحديث. يمكن تنظيم الكود تدريجياً إلى ملفات واضحة فقط إذا أثبتت اختبارات التطابق أن السلوك لم يتغير.

## 4. ما هو موجود بالفعل ولا يجوز تكراره أو بناؤه كميزة ثانية

اقرأ التنفيذ الحالي وتأكد منه، ثم حسّن نسخته الموجودة بدلاً من إنشاء نسخة موازية. المشروع يحتوي بالفعل على:

1. العربية RTL والإنجليزية LTR.
2. الوضع الفاتح والداكن وتكبير العرض والنص.
3. مستخدم، زائر، مزود فردي، شركة، إدارة وأدوار إدارية.
4. بوابة بدء واختيار نوع الدخول.
5. إرشادات أول استخدام مخصصة للأدوار وصور عمانية.
6. تسجيل المستخدم وحفظ الجلسة وتفضيلات الجهاز.
7. تسجيل المزود والشركة والتحقق والوثائق والصور.
8. بحث تدريجي يبدأ بالقسم ثم الخدمة ثم الفلاتر.
9. صفحة خدمات وشريط أقسام أفقي وبطاقات خدمات.
10. طلب مباشر متعدد الخطوات.
11. إظهار توفر الأقسام والخدمات من بيانات المزودين المؤهلين.
12. خريطة Leaflet وOpenStreetMap وموقع الجهاز وزر موقعي.
13. خصوصية ظهور موقع المزود وخيار `mapVisible`.
14. بطاقات مزودين وصور وتقييم وجودة وسعر يبدأ من وحالة توفر.
15. تفاصيل المزود والمفضلة والمشاركة.
16. ساحة طلبات عامة آمنة.
17. ترشيح مستخدم لمزود مطابق داخل طلب.
18. مطابقة دقيقة حسب القسم والخدمة والموقع والتوفر والاشتراك.
19. إرسال الطلب لأفضل 5 ثم التوسع إلى 10 وفق الشروط الحالية.
20. عروض مزودين وقوالب سعر ومدة ومقارنة عروض.
21. دورة حياة طلب وطلبات نشطة وملغاة ومؤرشفة.
22. محادثة مرتبطة بالطلب وبالحساب الصحيح.
23. رسائل نصية وصور وصوت وموقع داخل المحادثة.
24. موافقة مستقلة لمشاركة التواصل عبر `ContactConsentService`.
25. إشعارات مجمعة وروابط عميقة للسجل أو المحادثة المقصودة.
26. لوحة مزود وحالة متاح أو مشغول أو غير متاح.
27. مركز فرص عمل للمزود.
28. تقويم أسبوعي للمزود.
29. ملف المزود وصور الأعمال والفيديو وعرض قبل وبعد.
30. اختيار الصورة الرئيسية وقص الصور وضغطها وفق التنفيذ الحالي.
31. نظام جودة وتقييمات وشكاوى.
32. نقاط ولاء للمستخدم.
33. خمس باقات إنتاجية فقط وخدمات `SubscriptionService` و`EntitlementService`.
34. فترة تأسيس وتجديد وترقية وسماح وانتهاء وإيقاف ظهور دون حذف الحساب.
35. طبقة دفع وOTP جاهزة للربط ولا تدّعي النجاح دون مفاتيح حقيقية.
36. لوحة إدارة فيها طلبات ومزودون وشركات واشتراكات ومالية وإعلانات وتقارير وجودة.
37. Audit Log للعمليات الحساسة.
38. سياسات خصوصية وجودة وشروط وإلغاء بالعربية والإنجليزية.
39. تقارير CSV وDOCX وطباعة PDF.
40. PWA وManifest وService Worker وكاش بإصدارات.
41. Data Adapter وتوافق مع LocalStorage القديم وعزل الزائر.
42. اختبارات 320 و375 و390 و430 بكسل وسطح المكتب.

ممنوع إنشاء مساعد جديد منفصل، أو بحث ثانٍ، أو خريطة ثانية، أو لوحة إشعارات ثانية، أو مركز طلبات مكرر، أو نظام باقات موازٍ. حسّن المصدر الحالي فقط.

## 5. الملاحظات البصرية المثبتة من اللقطات الحالية

التقط لقطات جديدة بنفسك قبل التعديل، لكن اعتبر هذه المشكلات نقطة بداية مؤكدة:

1. بوابة البدء تحتوي فراغاً علوياً كبيراً وكتلة مركزية أطول من الحاجة.
2. بعض البطاقات تعتمد حدوداً كثيرة ومتساوية القوة، فتبدو الصفحة كشبكة صناديق بدل منتج حي.
3. ألوان الأسطح فاتحة ومتقاربة جداً، فتضعف الأولوية البصرية رغم أن التباين النصي مقبول غالباً.
4. أيقونات خدمات عديدة مكررة أو عامة مثل النجمة والمفتاح بدل أيقونة خاصة بالتخصص.
5. شريط تبويبات المزود يُظهر جزءاً مقصوصاً من تبويب عند عرض 320 بكسل، حتى لو كان قابلاً للسحب تقنياً.
6. شريط تبويبات الإدارة كثيف ويُقص عند الحواف ولا يثبت العنصر النشط في مجال الرؤية دائماً.
7. لوحة المزود طويلة وتعرض التقويم والقوالب والشكاوى والصور والاشتراك دفعة واحدة.
8. نموذج ملف المزود والوسائط يستخدم عناصر رفع الملفات الأصلية للمتصفح، فتبدو قديمة وغير متناسقة.
9. شاشة الدردشة تظهر طبقات متعددة وخلفية الحساب وراءها، مع رأس وToast وشريط موافقة وردود سريعة ومحرر في مساحة واحدة.
10. بعض حالات الموقع تظهر منخفضة التباين أثناء المعالجة ولا توضّح هل الصفحة تحمل أم معطلة.
11. بعض الشاشات تضع أرقاماً أو إحصاءات كثيرة قبل الإجراء الأساسي.
12. واجهة الإدارة تستخدم مساحات وبطاقات كبيرة لمؤشرات بسيطة.
13. الأسهم النصية الصغيرة داخل قوائم الحساب غير متناسقة مع بقية الأيقونات.
14. الصور الإعلانية جيدة كنقطة بداية، لكن تحتاج نظام Focal Point وقص متجاوب يضمن عدم قطع الأشخاص أو النص.
15. الصور الكبيرة داخل الدردشة تجعل السياق أطول من اللازم ولا تملك معاينة مصغرة متدرجة بوضوح.
16. تصميم شاشة الخدمات أصبح سليماً وظيفياً، لكنه يحتاج أيقونات متخصصة وهرمية أقل اعتماداً على الحدود.
17. البحث التدريجي جيد كأساس، ويجب تحسين كثافته لا استبداله.
18. بطاقة المزود الحالية المحسنة جيدة كأساس: الصورة منفصلة عن النص، فلا تعد إلى تغطية الصورة بالنص والسعر.

## 6. مبادئ التصميم غير القابلة للتفاوض

1. الهاتف أولاً من عرض 320px، ثم 375 و390 و430، ثم اللوحي والكمبيوتر.
2. كل شاشة لها إجراء رئيسي واحد واضح.
3. لا تستخدم بطاقة لمجرد وضع كل جزء داخل إطار.
4. لا تستخدم بطاقات داخل بطاقات إلا عند ضرورة وظيفية واضحة.
5. اجعل المعلومات الثانوية منسدلة أو في Bottom Sheet أو شاشة تفاصيل.
6. استخدم الإفصاح التدريجي Progressive Disclosure.
7. حافظ على اتجاه القراءة الحقيقي لكل لغة، بما فيه الأسهم وترتيب الأزرار والحركة.
8. ميّز بين زر الإغلاق X وزر الرجوع؛ الإغلاق يغلق الطبقة فقط والرجوع يعيد الشاشة السابقة.
9. لا تجعل الشفافية تقلل وضوح النص. الشفافية للمادة أو الخلفية فقط، والنص يبقى صلباً عالي التباين.
10. لا تستخدم تدرجات لونية وزخارف كبديل عن بنية واضحة أو صورة حقيقية.
11. لا تستخدم Emoji كأيقونات أساسية.
12. استخدم SVG أو مجموعة الأيقونات الحالية بأسلوب موحد وسماكة واحدة.
13. كل عنصر لمسي لا يقل عن 44px على iOS ويفضل 48px على الويب، مع 8px بين الأهداف المتجاورة.
14. احترم `safe-area-inset-*` وDynamic Island ولوحة المفاتيح وشريط Safari.
15. لا يظهر زر أساسي خلف لوحة المفاتيح.
16. لا تستخدم أحجام عناوين ضخمة داخل لوحات أو نماذج.
17. لا تستخدم حروفاً سالبة التباعد، ولا تضغط العربية لتلائم إطاراً خاطئاً.
18. لا تقص النص بصمت. اسمح بسطرين ثم انتقل إلى تفاصيل عند الحاجة.
19. لا تحجب الصور بنصوص كبيرة. يسمح فقط بحالة صغيرة أو زر مفضلة فوق الصورة.
20. الحركة تشرح الانتقال ولا تستعرض نفسها.

## 7. منظومة التصميم الجديدة

أنشئ Design Tokens مركزية، ولا تكتب ألواناً ومسافات عشوائية داخل كل مكون.

### 7.1 الألوان

ابنِ لوحة مشتقة من الشعار:

- كحلي عميق للهوية والرؤوس المهمة.
- تركواز واضح للإجراء الرئيسي والحالات النشطة.
- أخضر ثقة للحالة الناجحة والمتاح.
- ذهبي محدود للتمييز والولاء والباقة البارزة فقط.
- أبيض وأس surfaces محايدة دافئة قليلاً، وليست رمادية زرقاء باهتة في كل مكان.

أنشئ متغيرات دلالية مثل:

```css
--brand-950
--brand-800
--brand-600
--brand-100
--brand-50
--accent-gold
--surface-page
--surface-raised
--surface-sunken
--surface-overlay
--text-primary
--text-secondary
--text-muted
--border-subtle
--border-strong
--success
--success-soft
--warning
--warning-soft
--danger
--danger-soft
--info
--focus-ring
--shadow-1
--shadow-2
```

ضع مجموعة مستقلة للوضع الداكن. اختبر تباين WCAG AA للنصوص والعناصر. لا تعتمد على الأخضر والأحمر وحدهما؛ أضف رمزاً وكلمة.

### 7.2 المسافات والحواف

- شبكة 8px: `4, 8, 12, 16, 24, 32, 40, 48`.
- نصف قطر البطاقة القياسي 8px.
- حقول الإدخال والأزرار 8px.
- Bottom Sheet وDialog يمكن أن يكونا 16px في الحافة العلوية فقط.
- لا تجعل كل شيء كبسولة.
- استخدم ظلين فقط بدرجات خفيفة، ولا تجعل الحدود والظل قويين معاً.

### 7.3 الخطوط

- استخدم خطاً عربياً واضحاً محلياً أو System Stack سريعاً، ولا تعتمد على تحميل خارجي يحجب الصفحة.
- ضع Scale ثابتاً: 12 و14 و16 و20 و24 و28 كحد معتاد.
- النص الأساسي 16px على الهاتف حيث يسمح السياق.
- Line-height عربية بين 1.45 و1.65 بحسب الحجم.
- اختبر تكبير النص 100% و125% و150% و200%.
- نسّق الأرقام والتواريخ والعملة وفق اللغة، مع `ر.ع` و`OMR` حسب السياق.

### 7.4 الأيقونات

- أنشئ قاموس أيقونات مركزي.
- استخدم معنى واحداً لكل أيقونة في التطبيق كله.
- وفر `aria-label` وTooltip للأيقونات غير المألوفة.
- استبدل الأسهم النصية بأيقونة Chevron تتغير مع RTL/LTR.
- أنشئ أيقونات خدمات دقيقة: كهرباء، سباكة، تكييف، تنظيف سجاد، نقل، حاسب، كاميرات، سيارات، تعليم، مقاولات وغيرها.
- لا تكرر أيقونة النجوم لعشر خدمات مختلفة.

## 8. إعادة بناء المعلومات والتنقل من دون تغيير الوظائف

### 8.1 المستخدم

اجعل التنقل السفلي بحد أقصى خمسة عناصر مستقرة:

- الرئيسية.
- الخدمات.
- الطلبات.
- البحث.
- حسابي.

حافظ على العنصر النشط وموضع التمرير. لا تفتح شاشتين للمهمة نفسها.

### 8.2 المزود والشركة

استبدل شريط التبويبات العلوي الطويل بتنقل هاتف واضح من أربعة عناصر أساسية:

- اليوم أو الرئيسية.
- الفرص والطلبات.
- المحادثات.
- الملف والحساب.

ضع التوفر كتحكم صغير واضح في الرأس، وليس تبويباً إضافياً مقصوصاً. اجمع الخدمات والتوفر والوسائط والاشتراك تحت الملف في أقسام قصيرة. على الكمبيوتر استخدم Sidebar ثابتاً.

### 8.3 الإدارة

على الهاتف:

- شريط علوي قصير للفئات الأهم.
- زر «المزيد» يفتح قائمة كل الأقسام.
- تثبيت العنصر النشط وإعادته إلى مجال الرؤية آلياً.
- شارات تنبيه على القسم نفسه.

على الكمبيوتر:

- Sidebar منظم حسب: التشغيل، المستخدمون، الإيراد، المحتوى، النظام.
- لا تستخدم شريطاً أفقياً بعشرة تبويبات.

### 8.4 الرجوع والإغلاق

- أنشئ أو استخدم Layer Stack موحداً.
- X يغلق Modal أو Sheet أو Map فقط.
- Back يعيد الشاشة السابقة مرة واحدة.
- احتفظ بالتبويب والفلاتر والتمرير والبطاقة المفتوحة.
- اختبر Deep Links وRefresh داخل صفحة الطلب والمحادثة والمزود.

## 9. تحسين كل شاشة

### 9.1 بوابة البدء

- قلل الفراغ العلوي.
- ضع الشعار والاسم والوعد المختصر في كتلة متوازنة داخل أول Viewport.
- خياران رئيسيان فقط: «أحتاج خدمة» و«أنا مزود أو شركة».
- «تصفح كزائر» واللغة في شريط علوي صغير.
- لا تكرر شرح الخصوصية؛ استخدم سطراً واحداً ورابطاً عند الحاجة.
- حافظ على الفخامة من خلال الصورة والفراغ المحسوب، لا من خلال Card عملاقة.

### 9.2 الصفحة الرئيسية للمستخدم

- الإعلان أول عنصر كما هو حالياً.
- الطلب المباشر هو CTA الوحيد القوي.
- ستة أقسام كحد أقصى مع «عرض الكل».
- لا تعِد إضافة «الأكثر طلباً» أو بحثاً مكرراً أو مزودين مناسبين إلى الرئيسية.
- أضف ملخص طلب نشط صغير فقط إذا لدى المستخدم طلب فعلي.
- استخدم Skeleton عند جلب الإعلان والطلب.

### 9.3 الإعلان

- نسبة موحدة تقريباً `16:7` أو نسبة تناسب الهاتف والكمبيوتر.
- دعم Focal Point يحدده الإداري.
- `object-fit: cover` مع Safe Zone للنص والوجوه.
- إذا لم يدخل الإداري عنواناً فلا تنشئ نصاً تلقائياً.
- معاينة هاتف وكمبيوتر قبل النشر.
- مؤشرات انتقال صغيرة، وإيقاف التدوير عند لمس المستخدم أو `prefers-reduced-motion`.
- أرشفة الإعلان وبيانات العميل والمبلغ والمدة من دون حذف السجل المالي.

### 9.4 الخدمات

- احتفظ بالشريط الأفقي القابل للسحب.
- أظهر أجزاء متوازنة من العنصر التالي لتوضيح إمكانية السحب، من دون قص النص.
- اجعل العنصر المختار واضحاً بلون وخلفية ورمز، وليس لون النص فقط.
- استخدم أيقونة حقيقية لكل تخصص.
- حافظ على كثافة ثلاث بطاقات حيث تتسع الأسماء بوضوح؛ عند 320px لا تصغّر النص تحت الحد، بل استخدم تكيفاً آمناً يثبته الاختبار.
- أزل الإطار الخارجي الكبير إذا لم يضف معنى، واستخدم مجموعات بعناوين واضحة.

### 9.5 البحث

- لا تغيّر منطق «القسم ثم الخدمة ثم الفلاتر».
- اجعل المرحلة التالية تظهر بحركة قصيرة بعد الاختيار.
- لا تعرض كل الفلاتر دفعة واحدة.
- زر تصفير واضح بعلامة X مع نص وصول.
- «البحث من الخريطة» يفتح خريطة مفيدة مرتبطة بالفلتر الحالي.
- بطاقة النتيجة تعرض: الصورة، الاسم، الخدمة، المنطقة، متاح أو وقت التوفر، الجودة، السعر الابتدائي وزر التفاصيل.
- اعرض Skeleton ثم Empty State مع إجراء بديل إذا لا توجد نتائج.

### 9.6 الطلب المباشر

حافظ على أربع خطوات فقط:

1. الخدمة الدقيقة.
2. الموقع.
3. تفاصيل مختصرة وصور اختيارية.
4. مراجعة وإرسال.

حسّنها كما يلي:

- Progress Indicator بسيط وواضح ويتقدم من اليمين في العربية.
- اجعل الاختيار المتاح أخضر مع رمز ونص، وغير المتاح أحمر هادئاً مع سبب.
- لا تسمح بمتابعة خدمة غير متاحة كأنها متاحة.
- لا تطلب الميزانية والوقت المرن في البداية؛ يناقشان بعد العرض كما هو التوجه الحالي.
- اختيار الموقع لا يعيد الخطوة الأولى.
- زر التالي Sticky فوق Safe Area.
- حفظ مسودة الطلب على الجهاز والحساب واستعادتها بعد الإغلاق.
- منع الإرسال المكرر عند الضغط السريع.
- شاشة نجاح حقيقية تحمل رقم الطلب وخطوة ما بعد الإرسال.

### 9.7 بطاقة المزود

- احتفظ بفصل الصورة عن النص.
- نسبة صورة موحدة مع Focal Point.
- فوق الصورة: المفضلة وحالة صغيرة فقط.
- أسفلها: الاسم، التخصص، المنطقة، الجودة، السعر وزر التفاصيل.
- لا تعرض علامة التوثيق العامة إذا كانت سياسة المنتج تجعلها للإدارة فقط.
- لا تعرض رقم الهاتف أو واتساب قبل الموافقة المطلوبة.
- ميّز الشركة عن الفرد بأسلوب خفيف لا يغير كل لون البطاقة.
- اعمل Placeholder جميل للصورة المفقودة لا يستخدم الشعار كصورة شخص.
- اجعل Share Card تحمل صورة واسم وتخصص ورابط عميق فعلياً.

### 9.8 تفاصيل المزود

- رأس مختصر بصورة وقرار سريع.
- تبويبات أو أقسام: نبذة، الأعمال، التقييمات، المناطق، التوفر.
- التقييمات داخل قائمة منسدلة أو شاشة فرعية.
- معرض الأعمال Lightbox مع تكبير وإغلاق وسحب.
- عرض قبل وبعد فعلي يدعم اللمس ولوحة المفاتيح.
- الفيديو لا يبدأ تلقائياً ولا يستهلك البيانات قبل الضغط.
- زر طلب الخدمة هو الإجراء الرئيسي.

### 9.9 الخريطة

- لا تُنشئ خريطة جديدة؛ حسّن الخريطة الحالية.
- لا تعرض نقطة افتراضية مضللة.
- ميّز موقع المستخدم وعلامات المزودين وClusters عند الكثافة.
- زر موقعي الحالي واضح ولا يغطي التحكم بالتكبير.
- Bottom Sheet مختصر عند اختيار مزود.
- احتفظ بفلتر الخدمة والموقع عند الإغلاق.
- لا تعرض موقعاً دقيقاً غير مسموح.
- وفر حالات رفض الإذن، الموقع غير متاح، عدم النتائج، الشبكة الضعيفة والتحميل.

### 9.10 الطلبات وساحة الطلبات

- اجعل «طلباتي» منقسمة إلى: نشطة، مكتملة، ملغاة أو مؤرشفة.
- اجعل الأقسام الثانوية قابلة للطي.
- بطاقة الطلب تعرض الحالة والخطوة التالية والوقت وآخر تحديث.
- ساحة الطلبات للمزود تستخدم فرزاً واضحاً: الأنسب، منطقتي، بلا عروض، الكل.
- لا تعرض فرصة غير مطابقة لتخصص المزود.
- ترشيح مزود من مستخدم آخر يبقى منفصلاً عن قبول العرض.
- أزل الطلب من الفرص بعد بدء التنفيذ، مع إبقائه في دورة حياة المزود الصحيحة.

### 9.11 العروض والمقارنة

- قوالب المزود تظهر كخيارات بصرية قصيرة، بحد أقصى ثلاثة في الصف حيث يتسع.
- يمكن للمزود تعديل القوالب وحذفها وترتيبها.
- لا تستخدم عبارة تقنية مثل `offer not found`.
- مقارنة المستخدم تعرض السعر والمدة والموعد والتقييم والموقع والشروط في جدول هاتف قابل للمسح البصري.
- بعد اختيار العرض، افتح الطلب والمحادثة الصحيحين ولا تترك زر مقارنة أو إعادة طلب غير مناسب.
- لا تفعّل اتصالاً مباشراً قبل قبول العرض وموافقة التواصل.

### 9.12 الإشعارات

- اجمع رسائل الشخص نفسه في محادثة واحدة مع عداد غير مقروء.
- اعرض الاسم، صورة صغيرة، آخر رسالة، الوقت، وعدد غير المقروء.
- افصل: المحادثات، الطلبات، النظام.
- الضغط يفتح السجل المحدد لا الصفحة العامة فقط.
- اجعل Badge مثبتاً على أيقونة الجرس نفسها.
- لا تعرض إشعارات المستخدم عند الدخول كزائر.
- لا تكرر طلب إذن Push؛ اطلبه في سياق مناسب مرة واحدة، ثم أدره من إعدادات الحساب.

### 9.13 المحادثة

- افتحها Full-screen أو كصفحة مستقلة على الهاتف؛ لا تُظهر الحساب أو الصفحة الخلفية من تحتها.
- رأس مختصر: اسم الطرف، حالة الطلب، تفاصيل قابلة للفتح، X أو Back صحيح.
- اجعل ملخص الطلب Accordion صغيراً.
- فقاعات واضحة، صور Thumbnail ثم Lightbox، صوت بتصميم موحد، وموقع كبطاقة خرائط.
- شريط الكتابة مثل تطبيقات المحادثة الحديثة: صورة، نص، ميكروفون، إرسال.
- أثناء التسجيل، اعرض الزمن والإلغاء والإرسال داخل Composer نفسه.
- لا تترك فراغاً منفصلاً لعبارة «جاري التسجيل».
- ضع الردود السريعة في Rail أفقي ولا تجعلها تحجب الرسائل.
- لا تجعل Toast يغطي اسم الطرف أو Composer.
- اعمل Optimistic UI مع حالة جارٍ الإرسال، تم، فشل وإعادة المحاولة.
- حدّث المحادثة تلقائياً بكفاءة، ولا تضع زر تحديث.
- حافظ على الموافقة على التواصل والخصوصية.
- أضف بحثاً داخل محادثة عند توفر رسائل كثيرة، من دون تعقيد الشاشة الأساسية.

### 9.14 حساب المستخدم

- رأس حساب مضغوط: الصورة والاسم والهاتف وأيقونة تعديل مألوفة.
- «إعدادات الحساب» تجمع الصورة والموقع والرمز واسترجاع الرمز.
- لا تكرر «تعديل» كنص إذا تكفي أيقونة مع وصف وصول.
- نقاط الولاء تظهر كشريط صغير قابل للفتح، لا Card كبيرة دائماً.
- اجعل حذف الحساب واضحاً ومتعمداً مع تحقق وآثار البيانات.
- حافظ على دخول المزود كخيار ثانٍ تحت رأس الحساب لمن لديه حساب مزود.

### 9.15 لوحة المزود

صمم أول Viewport كلوحة «اليوم»:

- حالتي: متاح أو مشغول أو غير متاح.
- الطلبات الجديدة.
- الموعد التالي.
- الرسائل غير المقروءة.
- اكتمال الملف كتنبيه صغير إذا ناقص.

انقل التفاصيل الثانوية إلى أقسام:

- التقويم.
- قوالب العروض.
- الأداء.
- الوسائط.
- الاشتراك.
- الشكاوى.

لا تعرض كل هذه الأقسام كاملة في الصفحة الرئيسية. استخدم Cards مختصرة تفتح تفاصيلها. احتفظ بالمؤشرات المفيدة: سرعة الرد، معدل الفوز، الطلبات المكتملة وسبب انخفاض الظهور، لكن اشرحها بلغة بسيطة.

### 9.16 ملف المزود والوسائط

- استبدل `<input type=file>` المرئي بواجهة رفع مخصصة مع بقاء Input أصلياً خلفها للوصول.
- بطاقات رفع: الصورة الشخصية، صور الأعمال، الوثائق، الفيديو، قبل وبعد.
- معاينة فورية، قص، تدوير، تكبير وتصغير، استبدال، حذف وترتيب.
- اختيار الصورة الرئيسية من الصور المسموح بها.
- حد واضح للحجم والنوع والعدد قبل الرفع.
- Progress حقيقي وضغط مناسب وعدم تجميد الصفحة.
- الوثائق تظهر للإدارة فقط عبر روابط موقعة.
- لا تضع رابط `example.com` أو بيانات تجريبية في الإنتاج.
- اعرض عدد المستخدم والمتبقي من الصور والخدمات والأقسام حسب الباقة.

### 9.17 الشركة

- لوحة تشبه المزود مع إضافة الفريق والفروع والصندوق المشترك.
- أدوار Owner وManager وStaff بصلاحيات خادمية.
- توجيه الطلب لعضو أو فرع.
- مؤشرات استعمال حدود الباقة.
- عدم كشف محادثة فرع لعضو لا يملك صلاحيتها.

### 9.18 الإدارة

أنشئ هرمية تشغيلية:

1. ما يحتاج إجراء الآن.
2. الطلبات والمحادثات والمخالفات.
3. المزودون والشركات والمستخدمون.
4. الاشتراكات والإيراد.
5. الإعلانات والمحتوى والخدمات والمناطق.
6. التقارير والنظام والإعدادات.

متطلبات الإدارة:

- كل بطاقة تنبيه تفتح السجل نفسه وتُبرزه.
- بحث وفلاتر محفوظة.
- إجراءات جماعية آمنة مع تأكيد.
- سبب إلزامي للإيقاف أو الرفض أو التعديل المالي.
- Audit Log قابل للبحث.
- إدارة الباقات والأسعار والمدة والحدود والتأخير والسماح.
- منح باقة مجانية كمكافأة مع سبب وتاريخ ومنفذ، من دون المرور بدفع وهمي.
- إيقاف ثم حذف مزود وفق سياسة احتفاظ واضحة.
- عدم انتحال المستخدم أو إرسال رسالة باسمه من شاشة الإدارة.
- الإدارة تستطيع المراجعة وإرسال تنبيه إداري واضح يحمل هوية الإدارة.
- شاشة جودة النظام تعرض حالة مفهومة ولا تعرض فشل اختبار اصطناعياً للمستخدم النهائي.

### 9.19 المالية والتقارير

- لوحة مالية تميز: الإيراد المحصل، المعلق، المسترد، الفاشل، المجاني الإداري.
- مؤشرات MRR وARR والتجديد والتوقف ومتوسط الإيراد لكل مزود.
- سجل مدفوعات وفواتير وكوبونات وحملات.
- لا تثق بمبلغ قادم من الواجهة.
- تقارير CSV بترميز UTF-8 BOM عند الحاجة لExcel العربي.
- DOCX بخط عربي واتجاه RTL صحيح.
- PDF عبر Print CSS أو مولد موثوق مع معاينة قبل التنزيل.
- اسم ملف واضح بالتاريخ ونوع التقرير.
- رأس يحمل شعار خدماتي وفترة التقرير ومصدره.
- لا تُحمّل رموزاً أو أحرفاً مشوهة.

## 10. الصور والهوية البصرية

1. افحص كل أصل بصري ولا تستبدل الجيد لمجرد التغيير.
2. استخدم Image Generation فقط عندما لا توجد صورة مناسبة أو توجد مشكلة قص أو تكرار واضحة.
3. أنشئ صوراً ذات معنى دقيق للخدمة، لا صوراً جوية أو مظلمة أو Stock غامضة.
4. اجعل الأشخاص عمانيين بملامح طبيعية محترمة ومن دون مبالغة.
5. العامل في ملابس العمل أو معدات السلامة لا يرتدي مصراً أو كمة فوق معدات العمل.
6. العميل العماني يمكن أن يظهر بالدشداشة والكمة أو المصر في سياق مناسب.
7. لا تكرر الوجه أو التكوين نفسه لكل حالة.
8. الإرشادات 1:1 عالية الدقة، والإعلانات بنسبة عريضة، وصور المزود 4:3 أو مربع حسب البطاقة.
9. احفظ WebP أو AVIF مع Fallback عند الحاجة، وحدد `width` و`height` لمنع CLS.
10. استخدم `srcset` و`loading=lazy` للصور خارج أول Viewport.
11. عرّف Focal Point لكل صورة قابلة للقص.
12. لا تضع نصاً عربياً داخل الصورة إذا يمكن عرضه HTML؛ هذا يحسن الترجمة والوصول.

## 11. الحركة والتغذية الراجعة

- مدة الحركة المعتادة 120 إلى 220ms.
- فتح Sheet من الأسفل وإغلاقه بعكس الحركة.
- انتقال اختيار المرحلة يوضح التقدم من دون تأخير.
- Press State وFocus State وDisabled State واضحة.
- Skeleton بدل Spinner في القوائم والبطاقات.
- Spinner صغير داخل الزر للعمليات القصيرة.
- Progress حقيقي للرفع والتقرير.
- نجاح وفشل بلغة مفهومة مع إجراء تصحيح.
- احترم `prefers-reduced-motion`.
- لا تستخدم Parallax أو Orbs أو مؤثرات Blur ثقيلة.

## 12. حالات الواجهة الملزمة

لكل شاشة أو عملية نفّذ:

- Initial.
- Loading.
- Skeleton.
- Empty.
- Partial data.
- Offline.
- Permission denied.
- Validation error.
- Server error بصيغة بشرية.
- Success.
- Retry.
- Disabled مع سبب.
- Expired session مع حفظ المسودة.

لا تعرض `HTTP`, `JSON`, `request failed`, `offer not found`, Traceback أو اسم جدول للمستخدم.

## 13. السرعة والأداء

1. قس قبل التعديل وبعده.
2. استهدف عند المئين 75:
   - LCP لا يزيد عن 2.5 ثانية.
   - INP لا يزيد عن 200ms.
   - CLS لا يزيد عن 0.1.
3. حافظ على جاهزية تفاعلية محلية تحت 3.5 ثانية، وقللها إن أمكن.
4. لا تحمّل الخرائط أو الفيديو أو معرض الصور قبل الحاجة.
5. ألغِ Event Listeners والموقتات عند إغلاق الشاشة.
6. امنع إعادة Render الكامل إذا تغير جزء صغير.
7. استخدم Debounce للبحث وAbortController للطلب السابق.
8. لا تعالج Base64 كبيرة في الخيط الرئيسي بلا ضرورة.
9. اضغط الصور في Worker أو على الخادم إن أمكن من دون إعادة منصة.
10. راقب الذاكرة عند فتح وإغلاق الخريطة والدردشة والمعارض عدة مرات.
11. خفف حجم `index.html` تدريجياً باستخراج القواميس والكتالوج والمنطق المستقل إلى ملفات واضحة، ملفاً واحداً في كل مرحلة مع اختبار تطابق. لا تنفذ تقسيم Monolith كاملاً دفعة واحدة.
12. لا تكسر `public/index.html`؛ أنشئ سكربت مزامنة وفحص Hash آلياً.
13. حدّث Service Worker بحذر، ونفّذ Activate وCache migration يمنع بقاء واجهة قديمة.
14. اختبر Slow 3G وOffline والعودة Online.

## 14. الوصول وسهولة الاستخدام

- WCAG 2.2 AA حيث ينطبق.
- ترتيب Focus منطقي.

## 15. التفرد الوظيفي الذي يضيف قيمة حقيقية

- هل حُلّت المشكلة؟

## 24. ما يجب حذفه أو دمجه

احذف فقط بعد إثبات أنه مكرر أو غير مستخدم، ولا تحذف بيانات:

- البحث المكرر خارج صفحة البحث.
- أي مساعد مستقل يكرر الطلب المباشر.
- أي «الأكثر طلباً» في الرئيسية إذا عاد من كود قديم.
- بطاقات شرح طويلة يمكن أن تصبح Hint قصيراً.
- زر تحديث يدوي لمحتوى يتحدث تلقائياً.
- زر رجوع داخلي إذا يوجد زر عالمي صحيح.
- نصوص «تجريبي»، «بدائي»، `example.com` أو رسائل تقنية.
- CSS overrides قديمة تخالف Design Tokens الجديدة، بعد التأكد من عدم الحاجة إليها للتوافق.
- Event handlers مكررة أو غير مستخدمة.
- صور Seed غير الملائمة من Build الإنتاج، مع إبقاء أدوات الإدارة لإضافة المحتوى الحقيقي.

لا تحذف:

- بيانات المستخدمين والمزودين والطلبات والمحادثات.
- مفاتيح توافق LocalStorage القديمة قبل Migration آمنة.
- مسارات GitHub Pages أو Render.
- أي ميزة قائمة تعمل إلا إذا استبدلتها بنسخة محسنة واختبرتها.
